# cerebro — 모니터링/알림 기능 설계 (FEATURE-MONITORING)

> **목적**: Pro의 핵심 리텐션·해자 기능인 "주제 모니터링 → 변화 알림"의 설계를 SSOT로 정의한다.
> **담당 역할**: Project Owner + Architect + Cyber Security
> **상태**: 설계 확정(구현은 M2/M3 — 하이브리드 계정 + 비용 가드 선행). 결정 근거: [ADR-0012](./adr/0012-monitoring-alerts.md). **cyber-security 적대적 검증(2026-06-27) 반영(§13).**
> **관련**: [ADR-0010](./adr/0010-bm-broad-first-intent-axis.md)(의도축 BM) · [ADR-0011](./adr/0011-llm-report-cache-and-licensing-gate.md)(캐시·약관) · [ADR-0008](./adr/0008-llm-usage-report.md)(LLM 리포트) · [DATA-SOURCING §8·§11](./DATA-SOURCING.md) · [SECURITY](./SECURITY.md) · [GTM §7.2·§6.1](./GTM.md)

## 1. 컨셉 & 가치
> **"검색은 한 번, 인사이트는 계속."** 사용자가 관심 대상을 구독하면, cerebro가 주기적으로 공개정보를 재수집해 **의미 있는 변화가 생겼을 때만** "무엇이 바뀌었고 + [내 의도] 관점에서 어떤 의미인지"를 알려준다.

- **의도축 정점 레버**: 일회성 검색 → **지속 의도**로 전환(리텐션·해자·고지불의향).
- **Pro 전용**(무료=없음). 리텐션·과금 엔진.
- 고지불의향 의도와 정합: 투자(공시·뉴스), 취업(채용·이슈), 브랜드/평판(마케터).

## 2. 범위 (In / Out)
**In(설계)**: 구독(주제+선택 의도) · 주기 체크(스케줄러) · 변화 감지(지문 diff) · LLM 델타 요약 · 이메일+인앱 알림 · 구독 관리/해지 · Pro 게이팅·한도 · PIPA·약관·비용 가드.
**Out(후속)**: 카카오 알림톡 · 근실시간 · 다이제스트 · 의도 필수화 · 변화 중요도 ML 스코어링.
**전제(선행)**: ① 하이브리드 계정(검색 익명·구독 로그인) = M2 · ② **전역 LLM 서킷 브레이커 + 예산캡**([BACKLOG](./BACKLOG.md) NOW#3, 기본 구현 머지 #48 — 모니터링 서브캡은 후속) · ③ 구독 CRUD `@fastify/rate-limit`.
**소스 제약([ADR-0014](./adr/0014-source-license-segmentation.md))**: 모니터링은 재수집·지문 저장·재가공이므로 **상업 OK 소스(Layer B: 위키백과·공공데이터·웹검색 등)만** 대상. **네이버·카카오는 무료 표시 전용이라 모니터링에서 제외**([DATA-SOURCING §11](./DATA-SOURCING.md)).

## 3. 구독 모델
- **단위 = 주제 + 선택 의도(활용 관점)**: 기본은 주제 단위, 원하면 의도(투자/취업/평판 등 9종)로 초점을 좁혀 그 관점 변화만 알림(노이즈↓·의도축 연결).
- **한도**: Pro 활성 구독 N개(예: 20). 무료 = 0. 계정당 캡 + 생성 rate-limit + 이상 패턴 탐지.
- **주기**: 기본 일 1회. (Pro 주기 선택 = 후속.)

## 4. 변화 감지 (핵심 — 약관·비용 정합)
약관상 스니펫 장기저장 금지(ADR-0011)이므로 **원문이 아니라 "지문"만 저장**하고, 쿼터·비용 폭증을 막기 위해 **상태·재수집·LLM 델타를 "주제(+의도)" 단위로 공유**한다(구독 단위 아님 — 동일 주제를 N명이 구독해도 **체크 1회**, 알림만 fan-out).

- **저장 지문**(주제+의도 키): 출처 URL 해시 집합 · 콘텐츠 지문(정규화 제목/스니펫 해시) · 직전 리포트 요약 해시 · `last_checked_at`. **스니펫 원문 미저장.**
- **체크 절차**: 재수집(fresh) → 새 지문 계산 → diff
  1. **새 출처**(URL 해시 신규, 특히 최근 뉴스/공시) → "변화 후보".
  2. 변화 후보 있을 때만 **LLM 1회**(주제당) → "변화 델타 요약 + [의도] 관점 의미" → 구독자에 fan-out.
  3. 변화 없으면 알림 없음 · LLM 미호출(**비용 0**).
- **fail-closed**: 재수집이 **빈 결과/일시 API 장애**면 "변화"로 간주하지 않고 체크 스킵(거짓 알림·"scrub됨" 정보누출 방지).
- **엔티티 앵커링**: 가능하면 wikidata 엔티티 id로 주제를 고정해 **동명이인 drift**(엉뚱한 주체 뉴스 매칭) 방지. drift 의심 시 알림 대신 저신뢰 플래그.
- **서버측 임의 URL fetch 금지**: 지문은 API 제공 스니펫으로만 계산. 본문 diff로 확장 시 반드시 `safeFetch`(호스트 화이트리스트) 경유.
- **중요도(MVP 규칙)**: 새 출처 유형·최신성 가중(news/공시 > blog/cafe), 의도별 우선순위(투자→news/공시).

## 5. 데이터 모델 (Supabase · 약관·PIPA 정합)
| 테이블 | 핵심 필드 | 비고 |
|---|---|---|
| `subscription` | id · **user_id** · query · type · angle? · cadence · active · created_at | 구독(사용자별). **RLS + 서버측 소유권 검증** |
| `topic_monitor_state` | **key = query+type+angle** · last_checked_at · source_url_hashes · content_fingerprint · last_report_hash | **주제 단위 공유**(구독자 N → state 1, H2 dedup). **스니펫 원문 없음** |
| `alert` | id · subscription_id · created_at · change_summary · new_source_urls[] · angle · read_at | change_summary = **추상·변형 요약**(스니펫 인용·근접복제 금지). 본문 저장 X·링크만. **보존기간 캡(예: 90일)**, 해지 시 즉시 삭제. RLS |

## 6. 알림 전달
- **이메일(주)** + **인앱 피드**. 이메일 = 저비용 provider(예: Resend 무료 3k/월). 카카오 알림톡 = 후속.
- **내용**: 변화 델타 요약 + 새 출처 **링크** + "cerebro에서 보기" CTA(**재유입 루프**). 빈도 캡(스팸 방지). 다이제스트=후속.
- **출력 인코딩(XSS)**: 외부 영향 동적값(출처 제목·URL)은 이메일=**HTML 엔티티 인코딩**(text-first, 외부 HTML 인라인 금지), 인앱=React 기본 이스케이프(`dangerouslySetInnerHTML` 금지). href는 **https/http만** + `rel="noopener noreferrer"`(SECURITY §5.1).
- **원클릭 해지 보안**: 토큰 = **HMAC 서명 · 고엔트로피 · 만료 · 단일목적(해지만, 데이터 미노출) · 1회성 · 상수시간 비교**. 메일 스캐너/프리페처 자동발화 방지를 위해 **확인 페이지→POST** 또는 RFC 8058 `List-Unsubscribe-Post`(멱등). **상태변경 GET 금지**(SECURITY §5.2).
- **클릭 트래킹**(KPI용 redirect)은 **대상 allowlist 또는 서명 토큰**으로 오픈리다이렉트 차단.
- 구독 관리 화면 + **원클릭 해지**(이메일 footer 포함).

## 7. 비용 · 컴플라이언스 가드
- **비용(선행 전제)**: **전역 LLM 서킷 브레이커 + 월 예산캡은 현재 코드에 없음**([BACKLOG](./BACKLOG.md) NOW#3). 이것 + **모니터링 전용 일일 LLM 서브캡**이 구현·검증된 **뒤에만** 모니터링 출시. 서브캡 도달 시 = 다음 윈도로 **연기**(알림 소실 금지, 최신성·중요도순 우선).
- **쿼터(H2)**: 주제 단위 공유 state로 재수집 dedup + **모니터링 쿼터 예산**(예: 일 쿼터 ≤30%, 라이브 검색과 격리) + **admission control**(예상 소비가 예산 초과 시 신규 구독 거부/대기). 배치 스케줄·백오프(DATA-SOURCING §5).
- **PIPA(지속 프로파일링)**:
  - 기존 공인·공개정보 게이트 통과 대상만. **구독 생성 게이트** = 생성 시 1회 검색 → 공인/비개인 엔티티로 resolve될 때만 허용(사인 거부).
  - **주기적 공인 재검증**(N일마다, 실패 시 자동 일시중지 + 고지) — 구독 후 사인화/지위변동 대응.
  - **민감 이벤트 필터**: 알림 트리거·요약 단계에 SECURITY §3.2 민감 카테고리(건강·수사/미확정 범죄보도 등)를 **별도 적용** — 검색 시점 PII 필터만으로 불충분, 해당 시 **fail-closed(알림 보류)**.
  - **보존기간 명시**(예: 알림 90일) · 해지/삭제 시 **즉시 하드삭제**(잊힐 권리) · 민감정보 추론 금지(리포트 가드 상속) · 감사 로그.
- **삭제요청 cascade**: subject가 삭제요청(privacy@)하면 **그 주체를 겨냥한 모든 구독 종료 + 알림 이력 purge + blocklist 재수집 차단**(SECURITY §3.4). 구독자 "해지"와 subject "삭제요청"은 **별개 경로**.
- **약관**: 지문만 저장(스니펫 미저장), 매 체크 fresh 재수집(갱신), change_summary는 추상 요약·링크만(원문 근접복제 금지).
- **rate-limit**: 구독 CRUD에 `@fastify/rate-limit` 적용(현재 server.ts 미등록) + 계정당 구독 캡 + 이상 패턴 탐지(1계정 무관 핫토픽 대량 = 의심).

## 8. Pro 게이팅 & 의도축 수익화
- 모니터링 = Pro. 무료 = 없음(GTM §7.2).
- **어느 의도를 모니터링하는지가 고지불의향 니치 신호**(Phase 0, GTM §6.1) → "투자 모니터링 팩" 등 후속 프리미엄 후보.

## 9. 사용자 플로우
검색 → 그래프 → **"이 주제 모니터링"**(로그인 필요·Pro·구독 생성 게이트) → (선택)의도 focus → 주기 → 확인 → 알림 수신(이메일/인앱) → 클릭 → 변화 반영 그래프로 **재유입** → 관리 화면에서 목록·해지.

## 10. 수용 기준 (AC, 구현 시)
- 구독 생성/조회/해지 동작. **타 사용자 구독 조회/해지 불가**(RLS·소유권 검증). 해지 토큰 위조·열거·GET 자동발화 불가.
- **구독자 "해지" ↔ subject "삭제요청"은 별개**: 삭제요청 시 해당 주체 **전 구독 cascade 종료 + 알림 purge + blocklist 차단**.
- 변화 없음/수집 오류/빈 결과 → **알림·LLM 호출 0(fail-closed)**. 새 뉴스 출처 생기면 주기 내 1회 델타 알림.
- **DB에 스니펫 원문이 저장되지 않음**(약관). `change_summary`에 원문 근접복제 없음. **비공인 주제는 구독 불가**(생성 게이트).
- 민감 이벤트(SECURITY §3.2)는 알림 보류. 이메일/인앱 출력 인코딩으로 XSS 0. 이메일 발송·원클릭 해지(POST) 동작.

## 11. KPI
Pro 구독률 · 알림 오픈율 · **알림→재방문율(리텐션 핵심)** · 해지율 · **의도별 구독 분포(니치 신호)**.

## 12. 미해결 / 후속
- 변화 중요도 스코어링 고도화(노이즈 저감) · 주기 선택 · 카카오 알림톡 · 다이제스트 이메일 · 근실시간(고WTP).
- **이메일 provider 선정 시**: 국외 위탁·이전 → PIPA 위탁/국외이전 고지·DPA 체크, 본문은 링크·최소정보만(민감 상세 금지).

## 13. 보안 검증 반영 (cyber-security 적대적 검토, 2026-06-27)
구현 게이트 전 반영해야 할 적대적 발견을 본 스펙에 통합했다.
- **머지 전 반영(High)**: H1 비용 서킷·서브캡 **선행 전제화**(§2·§7) · H2 **주제 단위 공유 state + 쿼터 예산 + admission control**(§4·§5·§7) · H3 **PIPA 5종**(구독 생성 게이트·주기 재검증·민감 이벤트 필터·보존기간·cascade, §7) · H4 **HMAC 해지 토큰·POST·RLS/IDOR**(§6·§10).
- **구현 시 반영(Med/Low)**: change_summary 추상화(§5) · 알림 XSS 인코딩(§6) · 구독 rate-limit·이상탐지(§3·§7) · fail-closed·엔티티 앵커링(§4) · 서버측 fetch 금지(§4) · 이메일 provider 위탁/국외이전·트래킹 리다이렉트(§6·§12).
- **구현 직전 재검증 게이트 유지**: 코드 작성 시 cyber-security 리뷰 1회 더.
