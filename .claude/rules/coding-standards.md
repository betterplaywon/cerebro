# Rule — 코딩 표준 (클린 코드 / 안티패턴 / 오버엔지니어링)

> 근거·전체 맥락: [`FOUNDATION-SPEC §4`](../../docs/foundation/FOUNDATION-SPEC.md). 이 파일은 **실행 체크리스트**.

## ✅ 항상
- 의미 있는 이름, 작은 함수(가이드 ~40줄), 단일 책임, 명시적 의존성.
- TypeScript `strict`. `any` 금지 → 불가피하면 `unknown` + 타입가드.
- 외부 경계(API 입력/응답, 환경변수, 수집 데이터)는 **zod 런타임 검증**.
- 부수효과 격리(순수 함수 우선), I/O는 경계 레이어로.
- 에러는 삼키지 않기. 사용자向 메시지와 로그를 분리. API는 일관된 에러 스키마.
- 주석은 "왜"를 설명(무엇/어떻게는 코드로). 죽은 코드·주석 처리 코드 커밋 금지.
- 보이스카웃: 만진 코드는 조금 더 깨끗하게(단, 스코프 폭주 금지).

## 🚫 안티패턴 (금지)
God object · 순환 의존 · 깊은 상속 · 매직 넘버/문자열 · 전역 가변 상태 · 복붙 중복 · 조기 추상화 · 거대 컴포넌트/함수 · prop drilling 남발 · `useEffect` 남용.

## ⚖️ 오버엔지니어링 경계 (YAGNI)
- 추상화는 **3번째 중복**에서. 가설적 미래 요구로 추상화 금지.
- 라이브러리 추가 전: "표준/내장으로 가능한가?" → 필요하면 가볍고 신뢰 가능한 것.
- 패턴(팩토리/전략 등)은 실제 문제를 풀 때만. 설정 가능성도 필요할 때만.
- 트레이드오프가 있는 결정은 `docs/adr/NNNN-제목.md`에 5~15줄로 기록(대안·근거·트레이드오프).

## 네이밍 (요약)
컴포넌트 파일 `PascalCase.tsx` · 기타 ts `kebab-case.ts` · 변수/함수 `camelCase` · 타입 `PascalCase` · 상수 `UPPER_SNAKE` · zod `XxxSchema` · 디렉토리 `kebab-case`.

## 커밋 전 자가 점검
- [ ] 린트·타입체크·테스트 통과
- [ ] 새 `any`/`console.log` 잔여물 없음
- [ ] 외부 입력 검증 존재
- [ ] 시크릿/하드코딩 비밀 없음
- [ ] 불필요한 추상화·의존성 추가 없음
